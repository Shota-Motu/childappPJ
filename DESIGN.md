# 📱 設計ドキュメント — 「1日1秒」子供の成長アルバムアプリ

> **コンセプト：「1日1秒で、子供の成長アルバムを作る」**
> 毎日たった1秒の動画を撮るだけ。カレンダーとアルバムで成長を振り返り、1年分をまとめて見返せる、シンプルな育児記録アプリ。
> （1 Second Everyday の和製版 × 子供の成長記録特化。Setlog のような最短操作で記録できる体験を目指す）

## 2. 保存方式の方針（ローカル vs クラウド）

**推奨：フェーズ1はローカル保存のみ。クラウドは「バックアップ機能」として後付け。**

| 観点 | ローカル | クラウド |
|---|---|---|
| ランニングコスト | ゼロ | ストレージ費が継続発生 |
| 実装コスト | 小 | 認証・同期・課金が必要 |
| リスク | 端末紛失・機種変更でデータ消失 | 障害・退会時の扱い |
| 容量 | 1秒動画 ≒ 1〜2MB → 1年で約 400〜700MB | — |

**設計上の備え：**
- 保存パス・メタデータ操作を `StorageService` インターフェースに抽象化し、将来 `CloudStorageService` を差し替え可能にする
- 機種変更対策として「全データを zip でエクスポート / インポート」機能をフェーズ2で提供（クラウド不要の救済策）
- クラウドを入れる場合はサブスク課金（例：月300円）とセットで検討 = コスト回収の目処が立ってから

### 2.1 クラウド導入時の技術方針（フェーズ3で採用判断）

- **バックエンド候補：Cloudflare R2 + Workers（第一候補）**
  下り転送量が無料で動画アプリと相性が良い。ストレージ $0.015/GB/月 → 1ユーザー年500MBなら月1円未満。
  対抗：Supabase（Auth/DB/Storage一体で開発が速いが転送量課金あり）、Firebase（下り課金が動画と相性悪め）
- **オブジェクトキー**：`{userId}/{childId}/{date}.mp4` — 1日1本の仕様がそのまま衝突解決になる
- **認証**：匿名認証 → メール / Apple / Google リンクの昇格パス（Firebase Auth / Supabase Auth）。
  SQLite が常に正（source of truth）なので、認証追加時に既存ユーザーのデータ移行は不要
- **課金**：App Store / Google Play のアプリ内サブスク必須（外部決済は規約リスク）。
  RevenueCat で領収書検証・エンタイトルメント管理（「クラウドバックアップ = プレミアム」）。無料枠から開始可
- **法務**：子供の顔動画を預かる時点でプライバシーポリシー・利用規約・退会時データ削除フローが必須
  （個人情報保護法、海外展開時は GDPR / COPPA）。技術より準備が重いため導入判断のチェックリストに含める

## 3. SNS化の方針

**推奨：SNS化しない。** 理由：

- 子供の顔動画という機微データを扱うため、公開機能はプライバシーリスクと運用負荷（通報対応・モデレーション）が跳ね上がる
- このアプリの価値は「自分（家族）のための記録」であり、他人の目を意識させると習慣化を阻害する
- 代替として **「家族間共有」**（招待した家族だけが閲覧できる）をフェーズ3候補に留める。共有はクラウド前提なので、クラウド導入判断とセットで検討する

## 4. 画面構成

```
(タブナビゲーション)
├── 📷 TodayScreen        … 今日の記録（起動時のデフォルト）
├── 🗓 CalendarScreen     … カレンダー
├── 🖼 AlbumScreen        … アルバム
└── ⚙️ SettingsScreen     … 設定

(モーダル / スタック)
├── CameraScreen          … 録画 → プレビュー → 確定
├── PlaybackScreen        … 単日再生（左右スワイプで前後日へ）
└── YearReviewScreen      … この1年の振り返り
```

### 4.1 TodayScreen（今日の記録）
**目的：毎日の記録を最短で**

- 起動して **2タップ以内に録画完了** が体験目標（起動 → 大きな録画ボタン → 自動で1秒撮影）
- 今日撮影済みなら：今日の動画サムネイル + 「撮り直す」ボタン
- 未撮影なら：大きな録画ボタン + 連続記録日数（ストリーク）表示
- 子供の月齢/年齢を表示（誕生日設定から自動計算）「生後 ◯ヶ月 ◯日」

### 4.2 CalendarScreen（カレンダー）
**目的：習慣化と、撮り逃した日の救済**

- 月表示カレンダー。撮影済みの日はサムネイル、未撮影日は空セル
- 今日以前の空セルをタップ → **端末のフォトライブラリからその日の動画/写真を取り込み**（撮り逃し救済。取り込んだ動画は先頭1秒にトリミング）
- 撮影済みセルをタップ → PlaybackScreen
- 月の記録率（例：「6月は 28/30 日記録」）を表示して習慣化を後押し

### 4.3 AlbumScreen（アルバム）
**目的：成長の振り返り・感情価値**

- 月ごとにグルーピングしたサムネイルグリッド（新しい月が上）
- 全期間ぶんを閲覧可能（SQLite インデックス + FlashList で仮想化スクロール）
- 上部に「この1年」ボタン → YearReviewScreen へ
- サムネイルタップで再生（PlaybackScreen、スワイプで連続閲覧）

### 4.4 YearReviewScreen（この1年）
**目的：このアプリ最大の「ごほうび体験」**

- 今年（現在年）の記録を月別に一覧表示
- 上部に「◯日分の記録」サマリー
- 各サムネイルをタップで再生
- **フェーズ2：「1年を連続再生」** — 記録済みの全動画を日付順にシームレス連続再生（プレイリスト方式なら ffmpeg 不要で実装可能）
- **フェーズ3：「1年ムービー書き出し」** — ffmpeg で1本の動画に連結し、カメラロールへ保存（誕生日に家族へ見せる、が最高の瞬間）

### 4.5 SettingsScreen（設定）
- リマインド通知 ON/OFF・時刻
- 子供のプロフィール（名前・誕生日）※月齢表示に使用
- テーマ（ライト / ダーク / システム）
- 言語（日本語 / 英語 / システム）
- データエクスポート（フェーズ2）
- ストレージ使用量の表示

## 5. 🎥 動画処理フロー

```
[録画開始]
   │ expo-camera で 1000ms 録画（maxDuration: 1 で自動停止）
   │ ※録画日はこの完了時点で確定（0時またぎで翌日扱いになるのを防ぐ）
   ▼
[ステージング]  cache/pending/{timestamp}{ext} へ移動
   │ ※拡張子は維持する（iOS は .mov、Android は .mp4）
   │
   ├─ プレビュー表示（ループ再生）
   │    ├─「この1秒を残す」→ 確定へ
   │    └─「撮り直す」→ プレイヤーから切り離してから一時ファイル削除 → 録画へ戻る
   ▼
[確定]
   1. documentDirectory/videos/{YYYY-MM-DD}_{version}{ext} へ移動
   2. expo-video-thumbnails でサムネイル生成
      → documentDirectory/thumbs/{YYYY-MM-DD}_{version}.jpg
      ※生成失敗は非致命（thumb_path='' で記録は成立させ、表示側でプレースホルダー）
   3. SQLite にレコード upsert
   4. upsert 成功後に旧バージョンのファイルを削除
   ▼
[画面離脱時]  未確定ファイルを削除。起動時にも pending/ を一括掃除
```

- **ファイル名はバージョン付き**（`{date}_{timestamp}`）：撮り直し時に必ず新しい URI になるため、
  プレイヤー・Image の URI キャッシュ問題と「再生中ファイルへの上書き」を構造的に回避できる。
  「1日1本」は DB の主キー (child_id, date) が保証する（ファイル名は保証に関与しない）
- 録画設定：720p / 30fps 程度（1秒なので画質優先でも容量は小さい）
- 撮影中はプログレスバー表示（1秒の視覚フィードバック）
- カメラの `onCameraReady` 前は録画ボタンを無効化（起動直後タップでの録画失敗を防ぐ）

## 6. データモデル（SQLite）

```sql
CREATE TABLE entries (
  child_id     INTEGER NOT NULL DEFAULT 1,  -- 複数の子供対応（フェーズ3）を先行確保
  date         TEXT NOT NULL,       -- 'YYYY-MM-DD'
  video_path   TEXT NOT NULL,       -- 相対パス（videos/{date}_{version}{ext}）
  thumb_path   TEXT NOT NULL DEFAULT '',  -- 生成失敗時は ''
  duration_ms  INTEGER NOT NULL DEFAULT 1000,
  source       TEXT NOT NULL DEFAULT 'camera',  -- 'camera' | 'import'
  content_hash TEXT,                    -- 同期の差分検出・重複排除用（撮影確定時に記録）
  sync_status  TEXT NOT NULL DEFAULT 'local',  -- 'local' | 'pending' | 'synced'
  remote_url   TEXT,                    -- クラウド上の場所（クラウド導入後に使用）
  created_at   TEXT NOT NULL,
  updated_at   TEXT NOT NULL,           -- 競合解決の基準（新しい方が勝つ）
  PRIMARY KEY (child_id, date)          -- 「1日1本」はここで保証
);

CREATE TABLE settings (
  key   TEXT PRIMARY KEY,  -- 'child_name', 'birthday', 'theme',
  value TEXT               -- 'locale', 'reminder_enabled', 'reminder_time'
);
```

- パスは**相対パスで保存**（iOS はアプリ更新でコンテナパスが変わるため）
- 月別グリッドは `WHERE date LIKE '2026-07-%'` で高速取得
- **スキーマは PRAGMA user_version による順序付きマイグレーションで管理**（将来の変更を安全に）
- **同期用カラム（content_hash / sync_status / remote_url / updated_at）は初期リリースから記録する。**
  後付けの ALTER TABLE では過去データの hash が欠落し、クラウド同期の初回移行で困るため
- **child_id も同じ理由で初期リリースから持つ**：フェーズ3の複数の子供対応時に
  主キー変更（テーブル再作成）という重い移行を避ける。それまでは固定値 1
- 撮り直しによる同日上書きは `updated_at` ベースの「新しい方が勝つ」を正式仕様とする
  （複数端末＝夫婦それぞれのスマホ対応時の競合解決も同ルールに乗せる）

## 11. 主なリスクと対策

| リスク | 対策 |
|---|---|
| 端末紛失でデータ全損 | **iOS：documentDirectory は iCloud 端末バックアップの対象**（ユーザーがバックアップ有効なら自動で守られる。要実機検証）。**Android は自動バックアップ対象外（25MB上限）のため zip エクスポートが生命線** → フェーズ2の最初に実装 |
| iOS アプリ更新でパスが変わる | DB には相対パス保存、起動時にベースパス解決 |
| 1秒録画の端末差（起動遅延等） | 実測で 1100ms 程度録画し、再生側で 1000ms にクランプ |
| 通知権限拒否 | 初回は撮影成功後の「良い体験の直後」に権限リクエスト |
| ストリーク切れによる離脱 | §12-D 参照（合計日数を主指標に） |

## 12. 改善提案（検討済み・未実装）

実装前にプロダクトオーナーの判断が必要なもの。優先度順。

### A.「間を外さない」撮影 — 3秒録画してベスト1秒を選ぶ（最重要・フェーズ2候補）

**課題**：子供は狙った瞬間に笑わない。「ボタンを押してから1秒」方式は、シャッターチャンスの
0.5秒前に気づいても間に合わず、撮り直し連発 → 記録が義務化して習慣が折れる最大要因になりうる。

**提案**：録画は 3秒（3000ms）行い、プレビューで「どの1秒を使うか」をスライダーで選ぶ。

- **動画ファイルは3秒のまま保存**し、選択した窓を `clip_start_ms` としてメタデータに記録
- 再生時は `clip_start_ms` から1秒だけ再生（プレイヤーのシーク制御のみ。**ffmpeg 不要**）
- フェーズ3の1年ムービー書き出し時に、ffmpeg で trim を適用して物理的に1秒へ切り出す
- 容量影響：1日 3〜6MB → 1年 1.1〜2.2GB。ローカル保存では許容range、クラウド原価も月数円レベル
- スキーマ影響：`entries` に `clip_start_ms INTEGER NOT NULL DEFAULT 0` を追加（マイグレーション1本）

デフォルトは先頭1秒（= 何もしなければ現行と同じ体験）。「選べる」が追加されるだけで操作は増えない。

### B. 今日の1秒を共有シートで送る（フェーズ2候補・実装約1日）

SNS 化はしない方針の代わりに、**OS の共有シート（expo-sharing）で動画ファイルをそのまま送れる**ようにする。
日本の育児家庭では「LINE の家族グループに送る」が支配的なユースケースで、
サーバー・アカウント・モデレーション一切不要のままアプリの口コミ導線にもなる。
置き場所は Today 画面の撮影完了ビューと再生画面。

### C. 写真しかない日の救済（フェーズ2のフォトライブラリ取り込みに含める）

撮り逃した日に動画がなく写真しかないケースは多い。`media_type` を持たせて写真も1日分として
受け入れ、再生・年間ムービーでは1秒静止表示する（書き出し時に ffmpeg で1秒クリップ化）。

### D. ストリークより「合計日数」を主役に（心理設計）

連続記録が切れた瞬間の喪失感は、育児アプリでは特に致命的（体調不良・入院などで数日空くのが普通）。
**主指標は「合計◯日分の思い出」「今月◯/30日」とし、ストリークは続いている間だけ添える**。
切れても失うものがない設計にする。※Today 画面は実装済み（合計日数を常時表示、🔥は2日以上のときのみ）

### E. ホーム画面ウィジェット（フェーズ3）

「今日まだ撮ってない」+ 昨年今日のサムネイルを表示するウィジェットは習慣化の最強装置。
iOS は expo-apple-targets + WidgetKit（ネイティブ実装が必要）。工数大のためフェーズ3。

### F. 課金の線引き（クラウド導入判断とセット）

- 無料：ローカル記録・閲覧・共有・年間連続再生（コア体験は全部無料 = 愛着を先に作る）
- プレミアム（月300〜500円）：クラウドバックアップ / 家族間共有 / 1年ムービーの高画質書き出し
- 「1年ムービー書き出し」自体を有料化するのは NG（最大のごほうび体験を人質にすると口コミが死ぬ）
